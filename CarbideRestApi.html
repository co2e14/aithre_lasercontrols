CARBIDE REST API documentation version v1

http://CARBIDE_IP_ADDRESS:20018/{version}

  • version: required(v1)

Introduction

The CARBIDE laser hosts a REST server, which greatly simplifies external control. This REST server is very capable, but not almighty, will only accept commands that mostly correspond to the CARBIDE User App logic (limited to mostly basic functions).

Since this API is a HTTP-based service, consuming it is easy from pretty much any platform, either it is LabView, .NET languages or Python.

Usage

The API is split into several endpoints:

  • /Basic. This endpoint contains all of the most frequently used functions. It basically replicated the main CARBIDE User App screen. Whatever one can press or see in the "Basic" tab of the User App, can also be found under this endpoint. In fact, User App itself utilizes this endpoint heavily.
  • /StretcherCompressor. All the compression stage-related functionality resides here.
  • /ExternalControl. Covers all the functionality related to the external connectors on the laser backplate (shot triggering, synchronization, custom TTL outputs, etc.).
  • /Advanced. Functionality contained here is not meant for everyday use. Moreover, some of the advanced functions, are only used by the User/Service Apps themselves and have no meaning for the user.
  • /Burst. Defines functionality related to CB3-Burst lasers. This endpoint is valid only for systems equipped with the Burst functionality. Ignore this endpoint if your laser is not equipped with the Burst functionality.

REST is all about resources, and each resource endpoint is made out of three parts: {server address}/{API version}/{local endpoint}.

  • {server address}. By default, CARBIDE lasers are shipped with a static IP 192.168.240.10. REST is always accessible via port 20018, so {server address} resource translates into "192.168.240.10:20018".
  • {API version}. Has a standard display form of "v1", "v2", v3", etc. It denotes the generation of the API; all endpoints added in one generation can only be removed in next one. For example, if "/ActualOutputEnergy" endpoint has been added to "v1", it will remain there forever. If, for whatever reasons,a decision will be made to rename it to "/EnergyOutputActual" (a highly unlikely change), it will only be changed in the next generation "v2". Even then, both generations "v1" and "v2" will operate side-by-side.
  • {local endpoint}. Simple readable text, which usually points out the function of the endpoint, like "ActualOutputPower" or "ExecutePreset".

Here are some examples:

  • 192.168.240.10:20018/v1/Basic/IsOutputEnabled
  • 192.168.240.10:20018/v1/Basic/Warnings
  • 192.168.240.10:20018/v1/Basic/CloseOutput

Any GET request can be issued using any browser, which is enough to monitor states. However, to practice PUT and POST requests, it is recommended to use a Chrome plugin (or a standalone installation) called Postman. It is also possible to use other softwares, such as Fiddler as well, but it is not usually used by the development team.

/StretcherCompressor/SaveAsShortestImpulseDuration post

post

CARBIDE ships with calibrated compression line. Sometimes, however, calibration reference point may drift (for example, stage moves slightly during transportation). In such cases, user may adjust the reference point.

  1. Move stage to a position of the shortest pulse.
  2. Issue this command. CARBIDE will take current stage position as the new reference point.
  3. All related data (like actual pulse duration) will be updated immediately.

/Advanced/ReduceLeak post

post

Initiates PP leak reducing algorithm, which tries to adjust timing parameters automatically. It usually takes up to 2 seconds. Note that results are susceptible to thermal drift; we recommend to run laser at target power for two hours prior to triggering leak reducing procedure, and then give an hour afterwards for the results to fully stabilize. Exact settle-down time highly depends on situation.
